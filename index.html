Interview wisdom
Tips that actually
make the difference
01 / 06
"Think out loud — interviewers are evaluating your process, not just the answer."
Before you type a single line, explain your approach. Walk through your reasoning, name the data structures you're considering, and ask clarifying questions. A candidate who thinks clearly but codes slowly beats one who codes fast but silently every time.
02 / 06
"Always clarify constraints before writing code — it signals senior-level thinking."
Ask about input size, edge cases, and whether the input is sorted. Can the array have duplicates? Can values be negative? These questions take 60 seconds and prevent wasting 20 minutes on the wrong solution.
03 / 06
"Start with brute force, then optimise — never start with the optimal solution."
A working O(n²) solution you can explain beats an O(n) solution you can't justify. State the brute force first, get agreement, then improve. This shows methodical thinking and builds trust with the interviewer.
04 / 06
"Time and space complexity analysis is not optional — it's expected at every level."
After coding, always state your time and space complexity unprompted. O(n), O(log n), O(n²) — know them, justify them. Interviewers at FAANG companies consider this a baseline requirement, not a bonus.
05 / 06
"Test your code on examples — including edge cases — before saying you're done."
Dry run your solution on the given example, then test empty input, a single element, and the maximum size. Catching your own bugs impresses interviewers far more than a perfect first attempt that breaks on edge cases.
06 / 06
"Pattern recognition is the real skill — learn 14 patterns and you can solve 90% of problems."
Sliding window, two pointers, fast & slow pointers, merge intervals, cyclic sort, BFS/DFS, dynamic programming, backtracking — master these patterns and most problems become recognisable variations rather than unknown puzzles.
